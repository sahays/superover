# LLD for Scene Analyzer Module

## 1. Introduction

The Scene Analyzer module is the first component in the **Core AI Analysis** layer of Project Super Over. Its primary
purpose is to take the processed video and audio files generated by the `video_processor` and `audio_extractor` modules,
send them to the Gemini AI, and retrieve a rich, scene-by-scene analysis of the content.

This tool acts as a bridge between the media processing and narrative synthesis layers, converting raw media files into
structured, analytical data.

## 2. Design Principles

- **Composability:** The tool is designed to work directly with the JSON manifest files produced by the upstream media
  processing tools, creating a seamless pipeline.
- **AI-Powered:** It leverages the Google Gemini Pro Vision model via the Python SDK to understand and deconstruct video
  and audio content.
- **Structured Output:** It produces a detailed, well-defined JSON output containing the analysis for each scene, making
  the data easy for downstream tools to consume.
- **Batch Processing:** It is designed to analyze multiple scenes (chunks/splits) from a single input manifest in one
  run.

## 3. Module Architecture

The module will be contained within a `scene_analyzer` directory.

```
scene_analyzer/
├── __init__.py
├── main.py         # CLI entry point
├── api.py          # API entry point (e.g., using FastAPI)
├── analyzer.py     # Core analysis logic (interacts with Gemini SDK)
└── models.py       # Pydantic models for input and output schemas
```

## 4. Detailed Component Breakdown

### 4.1. `models.py` - Data Models

This file defines the data structures for the API and the detailed JSON output schema.

**Pydantic Models:**

```python
from pydantic import BaseModel, Field
from typing import List, Optional

class SceneAnalysisRequest(BaseModel):
    # The primary input is the JSON report from a previous tool
    manifest_file_path: str

class CharacterDialogue(BaseModel):
    character_name: str = Field(description="The name of the character speaking.")
    dialogue: str = Field(description="The line of dialogue spoken by the character.")

class SceneAnalysis(BaseModel):
    start_timestamp_seconds: float = Field(description="Start time of the scene in seconds from the beginning of the original video.")
    end_timestamp_seconds: float = Field(description="End time of the scene in seconds from the beginning of the original video.")
    summary: str = Field(description="A concise, one-paragraph summary of the scene's plot and purpose.")
    setting: str = Field(description="A detailed description of the environment, e.g., 'A dimly lit, modern office at night, with rain visible on the windows.'")
    emotional_tone: str = Field(description="A single, descriptive term for the dominant emotion of the scene, e.g., 'Tense', 'Comedic', 'Somber'.")
    key_events: List[str] = Field(description="A list of the most important actions or plot developments that occur in this scene.")

    # Expanded Details
    characters_present: List[str] = Field(description="A list of all characters who are visibly present in the scene.")
    dialogue_transcript: List[CharacterDialogue] = Field(description="A structured, chronological transcript of the dialogue.")
    visible_objects: List[str] = Field(description="A list of significant objects, vehicles, or symbols visible in the scene, e.g., ['Laptop', 'Red sports car', 'Golden key'].")
    camera_movement: str = Field(description="A description of the camera work, e.g., 'Static shot', 'Slow pan from left to right', 'Handheld shaky cam'.")
    sound_design: List[str] = Field(description="A list of notable non-dialogue sounds, e.g., ['Distant siren', 'Ticking clock', 'Sudden gunshot'].")

class SceneAnalysisResult(BaseModel):
    source_manifest_path: str
    analyzed_scenes: List[SceneAnalysis]
    status: str
    message: str
    time_taken_seconds: float
```

### 4.2. `analyzer.py` - Core Logic

This file contains the core functionality for interacting with the Gemini SDK.

**Dependencies:**

- `google-generativeai`: The official Python SDK for the Gemini API.

**Core Logic:** The main function, `analyze_scenes`, will orchestrate the analysis.

1.  **Initialization:**
    - Configure the Gemini SDK with the API key from the `.env` file.
    - Initialize the `GenerativeModel` (e.g., `gemini-pro-vision`).
2.  **Input Processing:**
    - Read and parse the input `manifest_file_path`.
    - Identify the list of video chunks and their corresponding audio files.
3.  **Iterative Analysis:**
    - Loop through each video chunk from the manifest.
    - For each chunk: a. **Get Timestamps:** Extract start and end timestamps from the video chunk's metadata (using
      `ffprobe`). b. **Upload Files:** Use the Gemini SDK to upload the video chunk and its corresponding audio. c.
      **Construct Prompt:** Create a detailed prompt for the Gemini model, instructing it to return a JSON object
      matching the new, highly-detailed `SceneAnalysis` schema. d. **Send Prompt to Gemini:** Send the prompt and file
      handles to the model. e. **Receive and Parse Response:** Get the JSON response and validate it with the
      `SceneAnalysis` Pydantic model.
4.  **Aggregation and Output:**
    - Collect all the `SceneAnalysis` objects into a list.
    - Populate and return the final `SceneAnalysisResult` model.

**Example Gemini Prompt (Updated):**

```text
You are an expert film and television scene analyst. Your task is to deconstruct the provided video and audio files into a structured JSON format. Analyze the content and generate a single JSON object with the following exact structure and fields:
{
  "start_timestamp_seconds": <float>,
  "end_timestamp_seconds": <float>,
  "summary": "<A concise, one-paragraph summary of the scene's plot and purpose.>",
  "setting": "<A detailed description of the environment, e.g., 'A dimly lit, modern office at night, with rain visible on the windows.'>",
  "emotional_tone": "<A single, descriptive term for the dominant emotion of the scene, e.g., 'Tense', 'Romantic', 'Action-Packed', 'Comedic'.>",
  "key_events": ["<A list of the most important actions or plot developments that occur in this scene.>"],
  "characters_present": ["<A list of all characters who are visibly present in the scene.>"],
  "dialogue_transcript": [
    {"character_name": "<Name of the character speaking>", "dialogue": "<The line of dialogue spoken>"},
    {"character_name": "<Next character>", "dialogue": "<Their line>"}
  ],
  "visible_objects": ["<A list of significant objects, vehicles, or symbols visible in the scene, e.g., 'Laptop', 'Red sports car', 'Golden key'>"],
  "camera_movement": "<A description of the camera work, e.g., 'Static shot', 'Slow pan from left to right', 'Handheld shaky cam'>",
  "sound_design": ["<A list of notable non-dialogue sounds, e.g., 'Distant siren', 'Ticking clock', 'Sudden gunshot'>"]
}
Provide only the raw JSON object as a response, with no additional text or formatting.
```

### 4.3. `main.py` - CLI

**Dependencies:**

- `click`

**CLI Command:** The input is the JSON report from a previous step.

```bash
scene-analyzer ./video-output/large-video-file_report.json
```

**Functionality:**

- The CLI will take the path to a manifest file as its main argument.
- It will call `analyzer.analyze_scenes()`.
- It will save the output `SceneAnalysisResult` to a new JSON file in the same directory (e.g.,
  `large-video-file_report_analyzed.json`).
- It will print a confirmation message with the path to the new analysis file.

### 4.4. `api.py` - API

**Dependencies:**

- `fastapi`

**API Endpoint:**

- **Endpoint:** `POST /analyze`
- **Request Body (`SceneAnalysisRequest`):**
  ```json
  {
  	"manifest_file_path": "/path/to/video-output/report.json"
  }
  ```
- **Success Response (200 OK - `SceneAnalysisResult`):**
  - The full JSON object as defined by the `SceneAnalysisResult` model.

## 5. Error Handling

- **Invalid Manifest:** Handle errors from reading a non-existent or malformed JSON manifest file.
- **File Not Found:** The analyzer must handle cases where a file path listed in the manifest does not exist on disk.
- **Gemini API Errors:** Wrap all SDK calls in `try...except` blocks to handle API key errors, quota issues, connection
  problems, and model content safety violations.
- **Response Parsing:** Handle cases where the Gemini model does not return a valid JSON object, and implement a retry
  mechanism if necessary.
